As said above, I did that "because of time issues then" - it's been changed
now that I'm working at it again. But there's still one place where I
can hardly avoid a similar approach - that is when another process has to
replace the current one _seamlessly_. That's why I deinitialize most
everything except the window just before executing the new process, do a
fork there and have the child process keep the display for a couple seconds
so the newly started process can build its windows above it. This does work
quite well, and since the old process is torn down completely, it didn't
show any bad effects yet, even running this way for several weeks, even
months. The Linux system cleans up quite thourughly after a process is
gone. ;-)
But I might even try to have the child process terminate normally after
it's through - the chaining routine just was meant to serve for non Gooey
applications also. For Gooy^WGUI use, simply returning should suffice.
(If you'd want to have a look at the code, I can post it. It's not very
difficult. But I wonder if anyone can make use of it - it was created for
a very special purpose, after all.)
