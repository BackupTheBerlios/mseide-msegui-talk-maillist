Then probably not all went well. ;-)

> I understand that the i18n mechanism is really meant for translating
> labelling strings and other text, which I don't need (yet), but you said
> it could be used for other modifications, too. Is that not supported any
> more?

MSEi18n can be used in order to build different *.mfm files with different
property values based on common master versions. The *.mfm files are
compiled into resource so's or dll's which can be switched at runtime.
MSEi18n supports string and numeric values. A form designer in order to
change widget position and size is planned but difficult to implement.

> What should one use instead? Could this be solved using "skins"? (Do 
> skins support changing sizes of control elements including the whole
> window?)

tskincontroller has hooks where user code can control all aspects of loaded
components (onbeforeupdate, onafterupdate). I don't think tskincontroller
should be used for layout purposes.

> Or is there a mechanism similar to the IDE's way of building an 
> application's window reading the .mfm file which is also available at the
> user level? (This would probabely be the most elegant solution.)
> 
MSEi18n does this, components can be loaded from streams
(TStream.ReadComponent(), TReader.ReadRootComponent()).

> I don't mean to inconvenience you - if there's no easy solution, please
> just tell me and ignore my inquiry; I'll try to find some way myself then.

Suggestion: use automatically adjusting layouts. Some possible elements:

- twidget.anchors.
- tspacer.
- tsplitter.
- tlayouter.
- tmseform.onchildscaled and several other event properties.
