Sieghard wrote:

> 
> As an aside, since quite some time I keep getting the compiler (linker,
> actually) message
>  '/usr/bin/ld: warning: link.res contains output sections;
>   did you forget -T?'.
> The compiled programs seem to run correctly, though. Above that, I was
> unable to get rid of this message by adding this option to any of the
> accessible command lines (mostly, it was rejected as not existing, and
> adding "linux" to the "T" as fpc itself would require had no effect).
> Does this have any significance?
> 
It is as designed by FPC. -k-T can not be used because FPC needs the default
linker script of the OS AFAIK.

> BTW, a couple remarks about the IDE window management:
>  I don't quite like the "habit" of leaving all windows displayed if a
> project is closed - clutters the screen and hides other windows one may
> want to access. (Wasn't that different formerly?)

No.

>  When exited with no project open, it starts out "at the root" (one's home
> directory) using an arbitrary layout;

It is not arbitrary but the layout of the MSEide *.mfm files at compile time
of MSEide.

> i.e. it does not remeber the layout 
> on closing. Not too much of a concern, but distracting.

The whole state information of MSEide with exception of the values
in 'Settings'-'Configure MSEide' is stored in the *.prj file. I suggest to
use a "universal" project if you don't want to setup different *.prj files
for different projects.

>  When I prepared the sample project, saving the project "as copy" didn't
> save any of the subordinate files (not even the main file), nor did it ask
> whether they should get new names.

'Project'-'Save as copy' stores the *.prj file only. MSEide does not know
which files belong to a project, this is a major difference to other IDE's.

> Saving the main file under a different 
> name did not save the form file under that name,

Works for me with form units. The main (program) file has no direct
relationship to *.mfm files.

> either, nor did explicit 
> renaming of this and subsequent saving modify all the pertinent entries -
> the .mfm file still contained references to the old form name.
> 
In order to manipulate settings in *.mfm files add the form unit
to 'Project'-'Tree'-'Pascal Units', classtype, instance variable name and
component name can be changed in the list.
