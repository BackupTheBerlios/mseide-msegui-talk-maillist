Julio Jiménez wrote:

> Is this experimental locate slow using only one key? or it's slow using
> more than one key? Why not keeping for compatibility the old locate sintax
> too (calling this new locate with the proper parameters)?
> 
Locate() is slow compared to tlocalindex.find(), there is no significant
difference for single key and multi key versions of locate().
Single key versions of locate have been removed because they had separate
search implementations for the same task and I like to have a single
implementation for a task because of less possibility of errors.
BTW, should I add an "isnull" parameter to locate() or should "isnull" be
removed in find()? Currently NIL in "avalues" denotes searching a NULL
field value, this could be done for find() also.
