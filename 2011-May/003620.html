Am Montag 16 Mai 2011, 13:42:42 schrieb Christian Nobel:
> On 16-05-2011 07:18, Martin Schreiber wrote:
> > Please use a timage instead a tdataimage, tdataimage is foreseen for
> > placement into twidgetgrid only.
> 
> However the tdataimage and timage behaves the same way when dropped on a
> form - but are there any drawbacks by using the tdataimage on a form?
> 
If you don't use the value property unnecessary overhead. If value property is 
used the image format data (PNG,JPG...) will be stored in memory which 
normally is not necessary.

> > Because there is a single image widget and bitmap instance for the grid
> > colum.
> 
> Sorry, still don't get.
> 
You expect that you can access a cell image by 
"
 tdataimage2.gridvalue[1].LoadFromfile(tfiledialog1.controller.filename);
"
This implies that there is for every image cell a tmaskedbitmap instance which 
is not true. There is a single tdataimage with a single tmaskedbitmap for the 
whole column.
tdataimage2.gridvalue[1] accesses the imagedata for the cell stored in a 
tgridansistringdatalist, same as 
 tstringedit2.gridvalue[1]:='xxx';
acesses the string data for the cell stored in a tgridmsestringdatalist.

> > It must be converted to a supported image data format (JPG,PNG...)
> 
> So I have a timage, and that has to be converted into a ansistring, but
> I can not find anything about this in the FPC documentation?
> 
Please use tmaskedbitmap.writetostring(). The necessary image format units 
(formatpngwrite, formatjpgwrite...) must be listed in uses.

> > Maybe it is better to use a tdbwidgetgrid, tdbdataimage and to store the
> > image data in a tmsebufdataset descendant
> 
> Sort of using the tmesbufdataset as a virtual database?
> 
tlocaldataset can be used as an in memory database for example.
