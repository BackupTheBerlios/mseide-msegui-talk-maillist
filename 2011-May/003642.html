On Wednesday 18 May 2011 14:39:25 IvankoB wrote:
> Me still have nasty troubles with proper assigning params to a
> parameterized query when it should be activated next a local mode query.
>
> A local_query.design_mode_active:=true => AfterOpen: a parametrized query
> with design_mode_inactive sets to active.
What means "sets to active"? you set it to active in code or it will be set to 
active by tfieldparamlink auto refresh?

> The parameterized_query.AfterOpen is called (scrolles & aggregates oneself
> & goes bookmarks/edits/posts some data to the local_query) twice. Possibly
> because of some param are established after opening the query ?
>
Possible if there are more than one tfieldparamlink with fplo_autorefresh are 
connected to a destdataset. Please use a single tfieldparamlink and 
tfieldparamlink.slaveparams (Warning: will be renamed to "destparams" soon) 
in order to setup additional parameters if there is a single source dataset.

> Is it 100% sure that all params are established for all design inactive
> queries on datamodule before any design active (local) query opens via
> RTTI so that the parameterized query opens and refreshes only once so that
> its AfteroOpen were safe to use ?
>
No, you are responsible for the correct opening order of datasets.

> PS:
> Also, tactivator on a datamodule seems not to open design active queries
> on the datamodule.
> Even the queries and the activator are active in design 
> time. Removing  the actiavtor from controller.activator allows the queries
> to open.

This is as designed. tactivator can be used to switch on/switch off the linked 
components by switching tactivator.active in object inspector. On runtime the 
linked components will be activated by the tactivator and not by the streamed 
t*dataset.active property.
