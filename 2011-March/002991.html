Hallo Graeme,

Du schriebst am Wed, 02 Mar 2011 12:05:43 +0200:

> (using either toolkit). That just shows that it is inherently the X11
> protocol issues that was designed 20+ years ago for very simple systems
> with black&white screens. Today's desktops require more, and faster
> throughput - which the X11 protocol simply can't do or keep up with.

Exactly the (lack of) availability of graphics performance by hardware made
it very urgent for the developers of X11 to address these issues thoroughly.
X11 was even designed to be used on terminals connected via _serial_
_lines_ to servers generating the graphics output commands, so there should
be any number of functions for speeding up and decoupling graphics output.
And today's hardware is so much faster than anything available back then
that even a clumsy protocol should be handled without any noticable delay.

> Make no mistake, there are some nice features of X11, but performance
> just isn't one of them.

But you _do_ make the mistake to ignore the features of X11 - it _is_
designed with performance as a central issue, just because of the
circumstances it was designed under.
What seems to generate the "performance hit" X11 is currently suffering is
neglect of its abilities to decouple generating and executing the drawing
commands and an overly optimistc reliance on the performance of current
hardware. Maybe that's worsened by sluggish implementation of these
features for current technology graphics cards, or even neglect to
implement them at all.
On the other hand, as I wrote before, I cannot really find X11 itself to be
so bad - so it might just only be neglect of the implementors of the
current desktop management systems.
