>> Humm... agree in part. If you have a simple list, ok... but if you
>> have a JOIN, complex filters, etc these classes lose their meaning.
>
>
> That was just a very simple example. Much more complex cases are
> easily handled too - and are the norm in real-word applications.
>
> Here is a more complex example. Say I want to pull out data for a
> report. The data comes from various tables. Again there are many ways
> to approach this, but I am pro OOP, so I would recreate a set of
> objects for each report I generate. This has many knock-on benefits.
> Here is how I would extract my data.
>
> var
>  lList: TReportDataList
>  lReportVistor: TReportVisitor.
> begin
>  lList := TReportDataList.Create;
>  lReportVisitor := TReportVisitor.Create;
>  lReportVisitor.Execute(lList);
>  //.... lList is now populate with report data objects
>
>
> TReportVisitor is a TtiVisitor descendant that contains the hard-coded
> SQL (or auto-generated SQL), knows how to create report data objects
> and populate them from the resultset. Internally database transactions
> are managed automatically by the OPF framework.
>
>
> Anyway, if you wanted to know more about tiOPF there are lots of
> information on the website.

I got it. But I see two problems here:
a) An explosion of objects and b) You still have SQL hard-coded.

a) If you have many, many reports, will have a ReportVisitor class for
each report?
b) For reasons performance, you still have SQL hard-coded... this is a
fact. But is really necessary create these overhead with objects to
show a report? See, I like your thoughts, but I am trying to be the
"devil's advocate".
