On 07/08/2011 09:24 AM, Martin Schreiber wrote:
> You should not store your local *.prj in the SCM repository. Instead export 
> the project settings without window information 
> by 'Project'-'Options'-'Settings', 'Save' and store that file in SCM.
> 'Load' imports the project settings into the current project.

So the IDE already supports that type of functionality, it is just a
manual process.

My project settings normally use the project file name as the suggested
name for the executable file name. So I guess I'll have to hard-code the
executable name then.

Couldn't the IDE simply load those session information from a separate
file when loading the project file? This would make things much simpler.
Lazarus and Delphi IDE's do this - so clearly the need has been justified.

Plus, your current idea still has problems. Say I use a "personal
project file" and a "clean project file". The latter being the one that
is committed to the SCM. Now I make some adjustments to the project
settings, add new units to the Project Tree window etc. Then export
those settings to the "clean project file".  Now none of the other
developers are going to see these changes, because they use there own
personal project file. Luckily I see there is a "load" option, which
they can use. But that still means I need to notify each develop to load
the new settings. Plus I'll have to remember to do this on my other
development PC's too.

Also what happens if they made changes to the project settings as well,
but hasn't committed yet. Loading settings from the "clean project" will
now override there local project setting changes - without warning??
