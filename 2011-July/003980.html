On Saturday, 30 July 2011, Marcos Douglas wrote:
>
> I got it. But I see two problems here:
> a) An explosion of objects

Those report objects are "lite" objects. They don't necessarily
descend from the same class as your business objects.
Comparing them to TDataset and its TField class instances, there is
really no difference in principle. Memory usage will be near
identical.

> and b) You still have SQL hard-coded.

Don't you write SQL for your reports too? As I mentioned hard coded
SQL is one option, you can normally let the framework generate the SQL
automatically as well. My personal choice is to write the SQL myself
so I can optimize it where possible. Not ideal, but I have created a
SQL editor tool that I use to maintain my databases, and write my SQL.
This tool has extra features for tiOPF as well. It can generate Object
Pascal source code from the SQL and resultset - reducing the amount of
pascal code I need to manually write by a considerable amount.


> a) If you have many, many reports, will have a ReportVisitor class for
> each report?

Just like you now have many many TDataset and TField instances for
each report. :-)

> b) For reasons performance, you still have SQL hard-coded... this is a
> fact. But is really necessary create these overhead with objects to
> show a report?

As I mentioned, the SQL can be auto generated. And even in your
current projects you too have many object instances. TDataset, TField
and friends. Your RAD style of programming is simply hiding those
objects from you.


> See, I like your thoughts, but I am trying to be the "devil's advocate".

No problems, I'm used to answers similar questions.
