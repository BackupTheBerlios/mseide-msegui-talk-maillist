>> Can you give me an example?
>
> See my previous reply... I use Linux, but the exact same can be done in
> Windows.
>
> I haven't written bat files since DOS 6 days, so you need to fill in the
> gaps. :-)
>
> The easiest is to manually create the output directory. eg:
>
> mkdir units
> mkdir units\i386-win32
>
>
>> I use this BAT to compile:
>  ---------------------------------------------------
>  @echo on
>  set myroot=%cd%
>  set myFPC=%myroot%\fpc\2.4.3
>  set PATH=%myFPC%\bin\i386-win32;%PATH%
>  cd mseide-msegui
>  ppc386.exe apps\ide\mseide.pas -Fulib\common\* -B
>  -Fulib\common\kernel\i386-win32 -Fi\lib\common\kernel
>  -FUunits\i386-win32
>  -omseide.exe
>  ---------------------------------------------------
>
> The last 4 lines must actually be in one long line in the batch file.
>
>
> This will output all all .ppu and .o files into 'units\i386-win32'
> directory, and create the mseide.exe in the root of your msegui directory
> (nice and easy to find).
>
> If you have done this once, you can remove the -B compiler option, and
> simply add a new entry somewhere in the top to clean out the
> 'units\i386-win32' directory before a compile (but not strictly needed).
>
>
> PS:
> I recommend you recusively delete all existing .ppu and .o and .a files out
> of your existing sources BEFORE you run the batch file. I don't know what
> is the DOS command to recursively do such a delete. Doing this means you
> have no stray (and possible old) compiled units lying around that could
> confuse the compiler - this is a good thing. ;)
>
>
> After this, all other projects based on MSEgui can simply reference the
> 'units\i386-win32' too (no need to have them recompiled for every project.
> This is done by adding the following to your MSEgui projects options.
>   -Fu<msegui>\units\i386-win32
>
> This has pros and cons. Quicker to compile your project, but then you need
> to remember to recompile MSEgui whenever you get a new MSEgui update. Also,
> less copies of compiled units lying around on your hard drive.
>

BTW, very interested this parameters:
# Default compiler settings
# -vi -O2 -CX -XX -B

# Compiler parameter explanation:
#  -O1  minor optimization and debugger friendly (a good/safe choice)
#  -O2  optimization (slow & unstable on 64-bit systems)
#  -XX  try to smartlink
#  -Xs  strip all debug symbols from executable
#  -B   build all modules
#  -OpATHLON64  optimize for AMD64
#  -OpPENTIUMM  optimize for Core-2-xxx processors
#  -ve  only report compiler errors
