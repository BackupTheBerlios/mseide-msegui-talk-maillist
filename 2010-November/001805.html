Hallo Martin,

Du schriebst am Sun, 7 Nov 2010 16:38:05 +0100:

> Seems as designed to me. The edit has the focus, in onkeyup you call 
> showmessage, in modal dialog with default button 'OK', you press Enter
> again, key_return keydown clicks the button and closes the message
> window, the edit widget has the focus again and receives the key_return
> keyup event. Press the OK button with mouse or Space key and there will
> be no second message window.

Maybe that's as designed, but it certainly is _not_ as expected by the
average user used to average applications. This is _not_ commonly done so,
and there's absolutely _no_ reason why anybody should expect a program to
react so.
And _no_ - requiring to use the mouse is _no_ work around.
E.g. - _what_ mouse, after all? There are applications where there is no
mouse and no work-alike device either.

> Do you remember, I did not want to add all this published event
> properties to the edit widgets because using them is not so easy as it
> seems? ;-) What do you want to achieve?

Simply a dialog working as expected by an average user.

BTW, why, in the first place, _does_ the edit widget act on the key_return
keyup event at all? Didn't it act on the keydown event already? Or did the
loss of focus change its behaviour? Then the resulting behaviour is wrong,
as a stray key release (keyup) after regaining focus cannot be assumed to
be meant for processing by this application window. It cannot be assumed to
be meant for this application window in any case if there wasn't an
unmistakably associated key press (keydown) before.
Trash the stray event, and everything will be fine.
