> > BTW, why, in the first place, _does_ the edit widget act on the
> > key_return keyup event at all?
> 
> It does not. It is Wahono's onkeyup handler. I long time was reluctant to
> add onkey* handlers to edit widgets but at the end i gave up because of
> the repeated user requests.

Well, I won't argue about the responsibility here.
It should be clear, though, that key processing in such an environment,
where both key press and release are handled separately with anything
possible to happen in between, is a rather complicated matter.
In any case, partial events that cannot be safely mapped to their
counterpart should be considered stray events that should not be handled at
all, possibly not even presented for processing.
Like the stray key release that may hit a window after focus has changed -
this might even happen if the window manager presented a focus change menu
to select a window by keyboard.

Just to call attention to one of the many pitfalls of a GUI...
