Hallo Martin,

Du schriebst am Sat, 27 Nov 2010 07:56:10 +0100:

> If a keydown event changes the focused widget it is logical for me that
> any further keyevents (the keyup of enter key for example) go to the
> current focused widget.

Maybe for you, because that's the easy way to implement.
But this behaviour can get you any number of weird and unwieldy effects,
because these key events are _not_ independent from one another. Any keyup
_has to be_ associated to a corresponding keydown, when it's occurring
within the same widget's context. A sole keydown may initiate some action,
but a sole keyup may never do so.
For illustration, imagine you have a program to delete any number of files
on your disk, and have it request a confirmation with the default answer
being "no" (also executed by pressing enter), but which will do the action
if it receives an "enter" key up event. You simply press "enter" to reject
the action - and the program deletes your files nontheless...
Of course, this is bad design - but what good is an event which cannot be
used for any sensible purpose? So either the keyup event should be removed
(most radical solution;), or there should be some mechanism to prevent a
lone keyup (usually occurring in conjunction with a focus change) to be
ignored. This can be some procedure to be implemented by the programmer,
if that's clearly stated in the documentation (possibly even illustrated by
an example), or it can be a mechanism implemented within the event handler
system.
