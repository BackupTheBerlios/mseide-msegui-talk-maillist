> Seems as designed to me. The edit has the focus, in onkeyup you call
> showmessage, in modal dialog with default button 'OK', you press Enter again,
> key_return keydown clicks the button and closes the message window, the edit
> widget has the focus again and receives the key_return keyup event. Press the
> OK button with mouse or Space key and there will be no second message window.

Commonly, end users like use keyboard to execute simple dialog, like
askyesno or showmessage.
>
> Do you remember, I did not want to add all this published event properties to
> the edit widgets because using them is not so easy as it seems? ;-)
> What do you want to achieve?
>
In my app (Acosys), especially in POS (poin of sales), I should create
interface that should be operated by cashier very simple and quickly.
They prefer to use keyboard instead mouse (sometimes without mouse).

 I create tstringedit to enter product serial number, if user type or
select one serial number, the dialog form is displayed, they should
choose "yes" or "no" (with keyboard), and for POS users always use
numpad from keyboard, enter is quickly to press than space bar.

I can't use ondataentered event, because sometimes they can select
only from dropdown or just press enter with default serial number from
default.

IMO, if enter key is pressed from askyesno, showmessage, etc, the
enter key shouldn't sent to next widget. the enter key is only for
replace mouse click functionality.
