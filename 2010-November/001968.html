Did you build this dialog yourself? So the input has to handled by some
event handlers in your dialog form code?
If so, did you think of marking the key code as _processed_ after your
handler has acted on it? I.e. does your handler set the "processed" flag
in the info record? If not, this might be the reason that the key event is
passed on until some handler does.
Does adding "es_processed" in "info.eventstate" affect this behaviour?

If this prevents the wrong action in your product code routine, this whole
discussion might (have) be(en) moot...

> > MSEgui has no OnKeypress event because accompanying characters are
> > stored in info parameter of key events.
> 
> I don't understand your reason clearly. Why onkeypress(var info:
> keyeventinfoty) cannot be used?

For one, because there _is_ _no_ "onkeypress" event, and for another
because all the information provided by a separate "onkeypress" is already
there in the parameters of "onkeydown", i.e. the ASCII character(s)
possibly produced by the key pressed (info.chars: msestring).

After all, all the Delphi "OnKeyPress" event does is to pass a modified key
parameter value reflecting the ASCII code of the key, with shift modifiers
(shift, control, possibly Alt Gr) applied, because that information is not
available in Delphi's "OnKeyDown" handler. If there is no such value,
"OnKeyPress" is not called at all.
