Du schriebst am Tue, 8 Feb 2011 10:11:40 +0100:

> parambyname('idvalue').asinteger :=  inttostr(tstringedit.value);
> 
> --> idvalue  not id  :)

And I would like to suggest to decide on a single place to put the type
conversion - ".asinteger" probably requires an integer type for assignment,
whereas "initostr" would provide a string value. That might not really fit,
so the following variants come to mind:

(I use "stringedit" a component name, because "tstringedit" is the type
name already and cannot be reused as something different.)

 parambyname ('idvalue').asinteger:= stringedit.value;

presumed that "tstringedit.value" has an integer type - but isn't the
"value" field of a "tstringedit" typed as _string_ anyway? So the really
syntacically correct statement might rather be something like

 parambyname ('idvalue').asinteger:=  strtoint (stringedit.value);
 ^unknown type           ^regarded    ^converts            ^string
                          as integer   string to integer
or simpler

 parambyname ('idvalue').asstring:= stringedit.value;

which still doesn't account for wrong input provoking a conversion
exception.

Sorry for the nit-picking - the original poster didn't exactly make the
impression of a master of FPC and component programming, so I think it
appropriate to analyse the syntactical requirements a little closer.
