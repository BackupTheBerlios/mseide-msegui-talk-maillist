Yes - and what _does_ that effect?
It does create a variable of that type to contain the reference to an
object of that class. Or, in foreshortened "oo-speak", it creates an object
of the type. Neither _are_ the type anymore, but refer to a specific
instance.

> and because he didn't write the instance name.

You could always resort to using a meta-naming scheme, e.g. using a
descriptive name in angle brackets ("<>") or so, just to avoid misleading
someone not thoroughly familiar with the conditions.
Though using the type name for an instance is a common misconduct here, it
doesn't become less misleading or more correct that way.

"Oo-speak" is sufficiently misleading in itself by trying to be overly
figurative that it can only gain from any attempt to be precise.

(E.g. a specifically preposterously figurative saying is using the term
"sending a message" for the action of calling a - virtual, i.e. vectored -
method. It's also difficult for newcomers to grasp the fact that an object
variable _only_ comprises the space for its data, but _not_ for its methods
- they are taken from the centrally defined class code.
There's much more in "object orientation" that's described too figuratively,
obscuring the information about its workings and thereby quite often
leading even skilled programmers to ridiculously inefficient constructs.)
