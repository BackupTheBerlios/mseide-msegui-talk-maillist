This is what I wrote before:

I got to send a follow-up to my previous message reporting success with the
window placement problem with openbox.

Although this issue has been solved with the new trunk (and, according to
your writing, peraps some versions before), there's a new issue with the
i18n utility.

The simple fact is, it doesn't work anymore, the program just hangs.

Taking a closer look with the debugger, one can see that the initialization
method calls it self recursively even before the program has a chance to
reset the event vector. Of course, this is still called by
"oneventloopstart" only, as you recommended and as it worked with the
release version. But with this version, it behaves here just as formerly
when called through "oncreate".

Sadly, an attempt to set the coordiantes, font sizes and similar things
using a simple parser for .mfm files wasn't successful as far, as the font
size settings didn't work correctly - it looks a bit like inheritance is
broken with these. Even though I _did_ select for component base type, the
sizes set by my routine were disregarded by the ancestor component (e.g. a
tbutton component didn't obey to a font size set if cast to tcustombutton.)
It looks like these have to be selected to exactly the component's real
type, because the font setters all seem not to be virtual methods.
