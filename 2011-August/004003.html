> @Martin
> 
> I was curious about your design used in Menus and MenuItems. I follow a
> similar design to Delphi, in the sense that fpGUI has TMainMenu,
> TPopupMenu and TMenuItem classes. I don't by default have TMenuItem
> variable references for each menu item though - only for those I know I
> need to change at a later point.
> 
> So if you have an array of menu items (no variable references), how does
> the developer find a specific menu item so they can maybe disable that
> item depending on some application state? Must they search for the
> Caption of the menu item? But isn't that a bit unreliable depending on
> translations or in the case of dynamic menu entries etc?  Or is there
> some why you can also store references to a menu items?
> 
tmenuitem has a name property which can be used if wanted and
"
   function itembyname(const name: string): tmenuitem;
   function itembynames(const names: array of string): tmenuitem;
"
functions. itembynames() searches a submenu tree.

But normally one would not access a menu item but the connected taction 
component which often is placed in a datamodule.  Attached for example the 
action datamodule of MSEide.

> Also what do you find more of an advantage in your design compared to
> Delphi?
> 
There is less streaming overhead and it is possible to manage the menu items 
in object inspector, no need for a decdicated menu editor.
