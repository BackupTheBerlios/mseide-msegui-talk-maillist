I don't mean the usage of utf-16 always being the string encoding, I
meant the name of the string type: UnicodeString

By all definitions, a unicode string can be UTF-8, UTF-16, UTF-16BE,
UTF-16LE, UTF-32, UTF-32BE, UTF-32LE

So Borland's "ideal" of staying that a type named "unicodestring" is now
always UTF-16, is a very bad definition of a Unicode string - thus it is
just plain wrong.

Instead they should have maybe created a string type called: UTF16String
This would have been a more accurate statement, and clearly tell the
developer what encoding that string type stores.

What grinds on me even more, is that FPC so blindly follows Delphi, and
simply ignores the fact of a very wrongly named string type.
