On Wednesday 19 January 2011 10:41:40 Graeme Geldenhuys wrote:
> Op 2011-01-19 11:19, Martin Schreiber het geskryf:
> > Without -Fcutf8 every string constant assignment to a UnicodeString
>
> That would explain it then. UnicodeString in FPC is always UTF-16
> (thanks to Delphi for making such a stupid decision, because Unicode
> doesn't automatically mean UTF-16), and neither fpGUI nor any of my
> projects use UTF-16 (or UnicodeString type) anywhere.
>
Delphi is not alone by using utf-16 internally. In my opinion it is the best 
and most convenient solution.

>
> So the -Fcutf8 is just a requirement for MSEgui (because internal
> encoding MSEgui is UCS2). I better update my MSEide build script then.
>
Not necessarily. MSEgui source uses ASCII (codepoints <= 127) only, so the FPC 
assumption of ISO 8859-1 encoding is OK.

> MSEgui still has a chance for corrupting strings though. Because UCS2 <>
> UTF-16 (yes the chances are small, but they are there).
>
The strings will not be corrupted, instead a codepoint bigger than the BMP 
MSEgui shows two unknown characters, the string memory representation will 
not be changed as long nobody inserts text between the two words of the 
surrogate pair. Same problem exists with decomposed characters in the BMP 
BTW.
