Well, these aren't really errors, just warnings. Someone meant to need an
auxiliary variable at some time, but when he indeed didn't, he forgot to
remove the declarations. Those units abound from such things...

> msebits.pas(501,15) Warning: Function result variable does not seem to
> initialized

This really doesn't mean anything - it may be caused because of the typing
error, which lets the compiler skip the initializing statements and leave it
with the result variable unassigned. If the primary error gets fixed, this
message should disappear.

> msebits.pas(501,16) Error: Variable identifier expected
> msebits.pas(504,16) Error: Variable identifier expected
> msebits.pas(540,15) Error: Variable identifier expected
> msebits.pas(543,15) Error: Variable identifier expected

These all relate to the array "int64exp10ar" defined in the interface of
the unit and even successfully used some lines above (480 & 483) and below.
This _also_ point strongly toward an inconsistency between the types
"currency" and "int64", since at all occasions where there is an error, a
variable of type "currency" is involved.
Couldn't you investigate a bit how these types are defined _exactly_ on ARM?
They seem to be principally incompatible. The routines working on "currency"
values seem to have to be completely reworked for this system.
If you can do without "currency", you might - for now - consider to just
deleting them.
