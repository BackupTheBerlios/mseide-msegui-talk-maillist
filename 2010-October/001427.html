> > Currently loadlangunit procedure can be problem when we store
> > langxxx.so different with application path.
...
> Maybe better to adjust LD_LIBRARY_PATH or LD_PRELOAD?

Aren't those meant as last and absolute last resorts, all other methods
failing?
For one, this requires a very ugly construct to call the program.
Also, this will make the loader search the language directory _first_ for
all libraries, as the paths specified therein take precedence over _all_
system paths.

> Absolute library paths on Linux look strange to me.

And yet they're common for dynamically loaded libraries, albeit often not
as uniquely determined ones, but with various common alternatives.
Nonteless, for a specially built library used by exactly one program only,
I'm inclined to see it fit to keep it at a place where this program expects
similar files also. This place does _not_ need to be the system library
directory, nor the program binary directory or a subdirectory thereof.

BTW, I posted a patch toward this issue in the message "beside". You're
free to use it, if you like.
