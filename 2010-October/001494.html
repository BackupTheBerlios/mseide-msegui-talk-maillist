  On 16.10.2010 11:50, Martin Schreiber wrote:
>   Agreed about your statement about event driven applications.
Thanks :)
>> _FPC _itself does not provide any Event Queue implementation by itself,
>> so Event Driven programming needs add-ons such as Lazarus or MSE.
> So why not use Lazarus
As said. Lazarus does a bad job with the Event Queue implementation. 
Firstly I feel that the way it is done (duplicating large code chunks 
for each Widget Type) is not good programming style and results in a 
manageability chaos, and secondly I do agree with the comments I got on 
a similar request some days ago in the German Lazarus Forum by Vincent 
that the Windowish Message is not a decent form event type for this 
issue, but a dedicated event providing for a flexible passing of queued 
parameters to the target event would be by far more sensible and  
"Object Pascal Style". (Here I feel the MSE Event Queue could be used or 
enhanced for that purpose.)
> or MSEgui?
This of course is a more personal cause. We have a couple of well 
trained Delphi programmer and need to port a set of large Delphi 
applications. It will be hard enough to re-train them on Lazarus and to 
do the ported programs in a way that they can be compiled as well with 
Delphi as with Lazarus. I fear that doing this with MSE will be not viable.
> I think FPC should concentrate on the
> compiler.
FPC already provides the RTL (compilable for lots of OSes and archs). As 
we agree that Event Driven programming is a core of Object Pascal, IMHO 
the Compiler / RTL  should provide this.

The syntax already provide the (Windowish - this not really nice) 
"procedure ... Message" construct for Event handling, but no means in 
the RTL that uses it.

If in future the syntax would be enhanced for a (Prism compatible) 
"parallel loop" / "future variable" construct (which I definitely vote 
for) the RTL would need to provide a thread pool to make this workable. 
Here I feel that en Event Queue might be a very sensible way to do the 
necessary synchronization.

> I forked most of the RTL (TThread for example which is a subject
> on fpc-pascal mailinglist at the moment) because the mainstream
> implementation did not fit the needs of MSEgui exactly. On the other hand
> functionality optimized for MSEgui will not please everybody.
This is very bad news indeed. A common code base for all FPC based 
systems would be very appropriate for an open source project. Is there 
no chance to re-unit the branches ?

I do know that TThread (WaitFor just seems to have been improved, 
Synchronize is known to not working perfectly ) is a weak spot with 
Lazarus. Maybe this is the result of a not perfect implementation in the 
RTL. So maybe Lazarus could benefit from your work on the RTL....

-Michael
