On Tuesday, 18. May 2010 16.17:55 Graeme Geldenhuys wrote:
>
> Yet another developer (and including myself) that would benefit from a
> "friend class concept" support in FPC. Where a Friend Class can access
> protected and maybe even private information from another class,
> without being declared in the same unit.

Or "Friend Units" where protected class members are visible. Sometimes if I 
mention such a construct on FPC mailing list I get the comment "Fix your 
flawed design." :-)
I think private should stay private and accessible in own unit only. I need 
access to private fields because I want to do things with FPC base classes 
which were not foreseen. A better solution is to do all ourselves from 
scratch. The biggest problem is TComponent which can not be forked without 
loosing the possibility to install foreign components into the IDE.

"Friend Units" would be convenient and better than local class alias because 
FPC AFAIK uses runtime resources for a definition like:
"
implementation
type
 txclass1 = class(txclass);
[...]
 txclass1(xclass).dosomething();
"
